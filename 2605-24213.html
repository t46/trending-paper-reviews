This paper makes evaluation harnesses a concrete software-engineering object, backed by a substantial GitHub issue corpus and a reusable workflow taxonomy. Its most consequential strength is the translation of scattered operational pain into an analyzable EvalEng vocabulary. The most important improvement direction is auditability — the replication package must expose the issue labels, prompts, model outputs, and scripts, and the main findings should be stress-tested with multi-label and severity-aware analyses.

Paper overview

Towards Evaluation Engineering studies ML evaluation harnesses as software systems in their own right: infrastructure that coordinates model invocation, dataset loading, metric computation, judge configuration, result aggregation, and reporting. The paper argues that these harnesses are now central to trustworthy ML evaluation because small operational failures can silently alter scores, rankings, and downstream conclusions.

The empirical core covers 57 evaluation harnesses, 19,638 GitHub issues, and 16,560 workflow-relevant issue labels. From this corpus, the authors extract a five-stage harness workflow model and a ten-category root-cause taxonomy, then analyze where reported friction concentrates across workflow stages and failure causes.

The main descriptive finding is that specification-stage issues dominate the observed operational burden, with unimplemented features, documentation gaps, and missing validation accounting for much of the reported difficulty. The paper is strongest when read as an empirical map of evaluation-infrastructure maintenance problems, and less settled when issue frequency is interpreted as a proxy for scientific risk or engineering priority.

Key strengths

Timely problem framing. The paper identifies evaluation harnesses as a distinct operational layer in ML evaluation infrastructure, which is especially relevant as LLM, multimodal, and agent evaluations increasingly depend on complex APIs, dynamic datasets, judge models, and reporting pipelines.

Clear RQ-driven empirical structure. The three research questions move naturally from workflow modeling to root-cause analysis to stage/cause interactions, giving the paper a readable descriptive arc and making the results easier for harness maintainers to translate into engineering priorities.

Substantial descriptive scale. A corpus spanning 57 harnesses and 16,560 workflow-relevant issue labels is large enough to support useful mapping claims about recurring operational friction in popular open-source evaluation infrastructure.

Reusable conceptual vocabulary. The EvalEng framing, five-stage workflow, and ten root-cause taxonomy are practical contributions. Even readers who disagree with some percentages can reuse the vocabulary for issue triage, design reviews, documentation audits, and maintenance checklists.

Transparent validity discussion. The paper explicitly acknowledges important limitations, including issue-reporting bias, the mismatch between frequency and severity, single-label simplification, and the focus on popular open-source projects. That candor makes the descriptive claims easier to calibrate.

Key concerns and constructive suggestions

Replication artifact traceability is the central weakness. The manuscript states that annotations, prompts, and labels are available, but the cited repository appears to correspond to a different foundation-model leaderboard survey rather than this paper’s issue labels and analysis pipeline. To address this, the authors should publish a paper-specific GitHub or Zenodo artifact containing issue IDs, repository snapshot metadata, manual labels, prompts, model versions, classifier outputs, aggregation scripts, and environment files.

Single-label classification may over-localize operational failures. Assigning one primary workflow stage and one root cause per issue can make messy failure chains look cleaner than they are, especially when configuration mistakes, validation gaps, documentation gaps, and external dependency changes interact. To address this, the authors should multi-label a representative subset and report how RQ2/RQ3 distributions change under primary-label versus multi-label interpretations.

Issue frequency is not the same as impact. The paper’s practical implications lean heavily on issue counts, but rare scoring bugs or judge-configuration errors may carry more scientific consequence than common documentation questions or feature requests. To address this, the analysis should add severity proxies such as closure time, maintainer labels, comments, reactions, affected benchmarks, exposure-normalized counts, or targeted manual severity annotation.

Uncertainty around generated distributions is underreported. The paper reports many percentages derived from classifier-scaled labeling, but readers need to know how stable the headline distributions are under sample variation, classifier confidence, and human/model disagreement. To address this, the authors should bootstrap issue samples, propagate classifier uncertainty, and compare human-only sample distributions against the LLM-scaled distributions.

Novelty boundaries need sharper positioning. EvalEng is a useful framing, but the manuscript should more directly distinguish evaluation harnesses from related work on leaderboards, LBOps, MLOps, benchmark frameworks, and testing infrastructure. To address this, the authors should add a compact comparison table that states what is shared, what is specialized, and which claims are unique to harnesses rather than evaluation infrastructure more broadly.

What would most improve this paper

1. Release a correct replication package. Publish the harness list, issue IDs, repository snapshots, labels, prompts, classifier configuration, raw model outputs, aggregation scripts, and environment files so the central empirical claims are auditable.

2. Add classification sensitivity analyses. Multi-label a representative subset, compare single-label and multi-label distributions, and report disagreement cases so the stage/root-cause conclusions do not depend too heavily on one labeling design.

3. Separate frequency, severity, and exposure. Add severity proxies, per-repository normalization, comment and closure-time analyses, and maintainer-priority labels to make the implications more useful for real harness maintenance.

4. Strengthen novelty positioning. Add an explicit comparison against LBOps, MLOps, benchmark frameworks, and software-testing infrastructure to clarify where EvalEng generalizes prior ideas and where it contributes a harness-specific lens.

5. Broaden external validity. Add lower-star repositories, private or industrial case studies, or a stratified follow-up sample to test whether the observed failure modes hold beyond popular open-source harnesses.
