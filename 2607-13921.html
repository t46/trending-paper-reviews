Generative compilation gives black-box code models early Rust feedback with formal guarantees and gains across seven models. Confidence is limited by two-run aggregates and unproved diagnostics; raw traces, uncertainty intervals, and diagnostic-quality tests are the key next steps.

Paper overview

Generative Compilation addresses a practical weakness of black-box code generation: standard compilers usually provide useful feedback only after an autoregressive model has finished producing a program. The paper introduces a sealor, a lightweight mostly syntax-guided transformation that turns a partial Rust-like program into a compilable complete program, allowing rustc or rust-analyzer to diagnose problems during generation.

The formal development separates two guarantees. A partial program that can still be completed should remain eligible, while partial programs that have entered genuine dead ends should remain detectable early enough to guide generation. These properties are proved for a core Rust-like calculus with Lean mechanization, and the approach is extended to real Rust with support for partial syntax, placeholders, control flow, and errors that depend on future code.

The evaluation covers seven frontier black-box and open-weight models on Translation and UpdatedAPI repository-level Rust tasks. Compared with standard post-generation feedback and a partial-compilation baseline, generative compilation reduces non-compiling outputs and improves functional correctness, while additional analyses examine detection position, diagnostics, timing, and the effect of restart-based feedback.

Key strengths

A crisp intervention point for black-box generation. The method uses ordinary compiler infrastructure to move semantic feedback into intermediate decoding without requiring white-box model access or a reimplemented Rust type system.

An elegant formal abstraction with a clear guarantee boundary. The sealor construction and its completeness/soundness split make precise why partial programs can be checked without incorrectly ruling out valid completions, and the Lean mechanization gives the core claims unusually strong support.

Promising evidence across models and task families. Seven models, repository-level Rust tasks, functional tests, timing measurements, and mechanism-level analyses provide a broader evaluation than a single-model compile-rate comparison.

Useful engineering reuse and public-artifact direction. Extending the approach to real Rust while delegating language semantics to rustc is practical, and the reported release of code, proofs, and benchmarks creates a foundation for independent reproduction.

Key concerns and constructive suggestions

Diagnostic faithfulness is not established. The formal guarantees cover the ok outcome, while the paper explicitly does not prove that an err diagnosis faithfully represents the partial program's true dead end. To address this, add a labeled benchmark for false-positive sealing artifacts, error localization, repair success, and human or agent usefulness, or prove a diagnostic-faithfulness approximation guarantee.

Uncertainty is difficult to judge from two runs. The main configurations are run twice and aggregate percentages are reported without per-run outcomes, standard deviations, confidence intervals, or effect sizes. To address this, release raw per-instance traces and seeds, report uncertainty intervals and exact effect sizes, and repeat the key comparisons across more independent runs.

The task distribution is narrow. Translation and UpdatedAPI are meaningful repository-level tests, but they do not establish performance on broader maintenance, security, interactive-agent, or out-of-distribution Rust workflows. To address this, add diverse repositories, task types, Rust versions, and independent evaluation harnesses.

Causal attribution remains entangled with the harness. Restart budgets, feedback prompts, check frequency, and timing may each mediate the observed gains, making it hard to isolate the contribution of early checking itself. To address this, run a factorial ablation covering partial compilation, early checking without sealing, generative compilation without restart, varied check frequencies, and prompt-format controls.

Positioning against constrained generation is incomplete. The lack of a full real-Rust constrained-decoding comparator is understandable, but it leaves the relative trade-offs unclear. To address this, compare with the strongest available semantic-prefix or grammar-based baseline and quantify implementation, runtime, and model-access costs.

Hand-written sealors may be brittle as Rust evolves. The construction is technically appealing but labor-intensive, and its maintenance burden across language versions is not yet measured. To address this, report version-portability and maintenance studies, then add automated rule derivation or validation tooling.

What would most improve this paper

1. Publish a reproducibility package with uncertainty. Include raw per-instance and per-run outputs, seeds, model/API/toolchain versions, pinned-container instructions, failure logs, confidence intervals, and exact effect sizes so the headline results can be independently reconstructed.

2. Add diagnostic-quality evaluation. Measure false-positive sealing artifacts, localization accuracy, repair success, and human or agent usefulness, making the practical meaning of compiler feedback as testable as the Boolean compile verdict.

3. Separate the mechanism from the interaction protocol. Use the proposed factorial ablation to isolate early checking, restart policy, check frequency, feedback wording, and model capability.

4. Broaden external validity. Evaluate more repositories, task types, Rust releases, open models, and independent harnesses, and include the strongest available constrained-generation comparator with engineering-cost measurements.

5. Reduce long-term maintenance risk. Automate sealor rule derivation and validation, and extend the formal and implementation framework beyond the current Rust setting.
