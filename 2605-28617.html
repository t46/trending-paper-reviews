LACUNA's strongest contribution is a clean typed-hole abstraction for safer code-as-action agents: model-generated code is checked against the expected type, lexical scope, and capability boundary before it can affect the environment. The most important next step is comparative evidence — matched baselines, uncertainty reporting, and public artifacts that make the safety and utility claims independently auditable.

Paper overview

LACUNA addresses a real tension in LLM-agent design: code-writing agents become more expressive when their generated code can shape control flow, context, and tool use, but that same expressivity expands the damage from prompt injection, wrong tool calls, and partial failures. The paper proposes a programming model where an agent action is written as a typed call, agent[T](task), whose implementation is supplied by an LLM at runtime and checked before execution.

The core idea is to treat generated agent code as a recursive program hole inside the host language. The surrounding program provides the expected type, visible names, and available authority; the generated code must satisfy those constraints before it runs. Code that fails these checks leaves the environment untouched, and compiler diagnostics can be fed back into the model for repair.

The prototype is built around Scala 3 compiler machinery and capture-checking ideas. The paper argues that this single abstraction can express ReAct-style loops, sub-agents, skills, parallel decomposition, and multi-model planning through ordinary host-language control flow, while using type, scope, and capability checks as a pre-execution safety layer.

The evaluation covers verifier test cases, BrowseComp-Plus, tau2-bench, and AgentDojo-derived prompt-injection settings. The resulting evidence supports feasibility and a meaningful safety story for type, scope, and authority errors, but it is not yet strong enough to settle broader comparative claims against other code-agent designs.

Key strengths

Unified typed-hole abstraction. The agent[T](task) primitive is compact and expressive. It lets familiar agent patterns — tool use, recursion, sub-agents, skills, parallel work, and multi-model planning — appear as ordinary program structure rather than as a separate agent runtime with bespoke control logic.

Concrete pre-execution safety property. LACUNA blocks generated snippets that refer to unavailable names, return the wrong shape, or violate capability constraints before they can change the environment. That is a clearer and more inspectable safety boundary than relying only on output parsing, runtime exceptions, or post-hoc monitoring.

Strong fit between method and host language. The design reuses Scala 3 type-checking and compiler diagnostics instead of inventing a fragile ad hoc validator. This makes the system contribution feel grounded: the paper shows how programming-language infrastructure can become part of an agent safety substrate.

Evaluation breadth across utility and security settings. The paper does not limit itself to toy examples. It tests verifier behavior, research-style browsing tasks, conversational tool tasks, and prompt-injection/capability scenarios, giving readers a useful initial map of where typed generated-code agents can help.

Honest treatment of limitations. The paper explicitly acknowledges that well-typed code is not necessarily correct code, that authority is only as narrow as the surrounding scope, that model coding ability matters, and that latency and resource costs remain practical constraints. This candor keeps the contribution interpretable as a safety substrate rather than a complete agent-safety solution.

Key concerns and constructive suggestions

Matched code-agent baselines are missing. The paper compares against native tool calling in some settings and against security-focused systems in attack settings, but it does not run CodeAct-style, RLM-style, ReAct, and native-tool baselines under identical model, tool, budget, and judge conditions. To address this, the authors should add a same-budget comparison table where LACUNA and the most relevant code-agent alternatives share the same models, tools, task limits, retry policies, and evaluation criteria.

Statistical uncertainty is underreported. The experimental tables mainly present point estimates, leaving readers unable to judge whether close differences are stable or practically meaningful. To address this, the authors should report bootstrap or binomial confidence intervals, per-task outcome distributions, retry counts, and failure-mode breakdowns for the main utility and security results.

Artifact availability gates independent validation. The paper describes an implementation, verifier suite, prompts, and evaluation harness, but no public repository is currently available in the reviewed materials. To address this, the authors should release the compiler branch, library code, benchmark adapters, prompts, scripts, and representative logs before relying on reproducibility or deployment-oriented claims.

The guarantee boundary can be overread. Type, scope, and capability checks can prevent important classes of invalid or over-authorized code, but they do not prove that the generated program chose the right in-scope tool, interpreted the task correctly, or achieved the user's semantic goal. To address this, the paper should add a front-loaded guarantees-and-non-guarantees box and pair the static checks with task-level validators, runtime monitors, or contract examples for high-stakes actions.

Portability and developer burden need sharper treatment. The design depends on static typing, effect or capability tracking, in-process recompilation, and access to live call-site context; it also relies on developers to expose only appropriate authority. To address this, the authors should provide a host-language portability checklist and ergonomic least-authority patterns that show how ordinary developers would scope tools and data without excessive boilerplate.

What would most improve this paper

1. Add matched baselines. Run LACUNA, native tool calling, CodeAct-style code actions, RLM-style REPL recursion, and ReAct under identical models, tools, budgets, retry policies, and judges. This would turn the comparison story from suggestive to directly interpretable.

2. Add uncertainty and failure distributions. Report confidence intervals, per-domain distributions, retry counts, compile-failure categories, semantic-failure categories, and raw task outcomes. This would let readers distinguish robust improvements from close or noisy differences.

3. Release the full artifact. Publish the compiler branch, Scala library, verifier tests, prompts, benchmark wrappers, scripts, and logs. This is the highest-leverage reproducibility step and would also make future baseline comparisons easier to audit.

4. Clarify guarantees and non-guarantees early. Add an explicit box in the introduction explaining what LACUNA checks before execution and what remains outside the static safety story. The paper should repeatedly frame the system as a safety substrate, not a semantic verifier.

5. Provide a portability and ergonomics appendix. Spell out the language features required to recreate LACUNA outside Scala and show concrete least-authority scoping patterns for real applications. This would help readers separate the general idea from the current prototype's host-language assumptions.
