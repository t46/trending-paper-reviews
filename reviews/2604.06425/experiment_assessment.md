# Experiment Assessment: Neural Computers (arXiv:2604.06425)

## Phase 3-3 — Experimental Design Assessment

---

## 1. Experimental Setup Summary

### 1.1 Overview

The paper reports 10 numbered experiments across two subsystems. Experiments 1–6 cover NC_CLIGen (terminal/CLI); Experiments 7–10 cover NC_GUIWorld (desktop GUI). Each experiment is color-coded, named, and produces one or two quantitative takeaways. All experiments are conducted in open-loop rollout mode: the model receives a ground-truth conditioning sequence (captions, first frames, or action logs) and generates video; generated frames are never fed back as model input.

### 1.2 Datasets

**CLIGen (General)**
- Source: Publicly available asciinema .cast terminal recordings
- Size: ~823,989 video streams (~1,100 hours)
- Domain: Diverse real-world terminal use (developer workflows)
- Preprocessing: Replayed with agg/ffmpeg at 15 FPS, normalized resolution, sensitive strings removed
- Captions: Three-tier (semantic/regular/detailed) generated by Llama 3.1 70B from terminal buffers
- Use: Training NC_CLIGen General; qualitative Experiment 1 visualizations

**CLIGen (Clean)**
- Source: Scripted VHS traces in Dockerized containers (~250k authored scripts)
- Retained: ~128k scripts (51.21%) after undescribed filtering
- Two subsets: ~78k "regular traces" (package installation, log filtering, REPL) + ~50k Python math validation traces
- Domain: Controlled, deterministic terminal interactions
- Captions: Derived directly from VHS scripts (scripted captions, not LLM-generated)
- Use: Experiments 2–6 (all CLIGen quantitative evaluation)

**GUIWorld**
- Source: Recorded Ubuntu 22.04 / XFCE4 / 1024x768 / 15 FPS desktop interactions
- Three splits:
  - Random Slow: ~1,000 hours (slow exploration, long pauses)
  - Random Fast: ~400 hours (denser cursor motion, typing bursts)
  - CUA supervised: ~110 hours (goal-directed trajectories from Claude CUA)
- Domain: Fixed app set (Firefox ESR, GIMP, VLC, VS Code, Calculator, Terminal, File Manager, Mahjongg)
- Action logs: Synchronized via xdotool; both raw-action and meta-action views stored
- Use: Experiments 7–10

### 1.3 Baselines

**NC_CLIGen baselines:**
- Wan2.1 (base model, no CLIGen fine-tuning): Table 5 (arithmetic probes); Table 2 implicitly (VAE reconstruction quality)
- Veo3.1 (Google, 2025): Table 5 (arithmetic probe accuracy: 2%)
- Sora2 (OpenAI, 2025): Table 5 (arithmetic probe accuracy: 71%)

**NC_GUIWorld baselines:**
- Untrained model (baseline_1): Table 8 (overall FVD/SSIM/LPIPS); Table 10 (action-driven metrics)
- External conditioning early-stopped (baseline_2): Table 10, flagged with footnote as rough reference (~50% training compute)
- Ablation conditions (not external baselines): contextual, residual modes compared against internal

**No inter-system comparisons are made between NC_CLIGen and NC_GUIWorld.**

### 1.4 Evaluation Metrics

**NC_CLIGen:**
- PSNR (dB): Peak Signal-to-Noise Ratio — pixel-level frame quality
- SSIM: Structural Similarity Index — perceptual quality
- Character accuracy: Levenshtein-distance-based, OCR via Tesseract on paired (ground-truth, generated) frames
- Exact-line accuracy: Fraction of ground-truth lines exactly matched in normalized generated text
- Arithmetic probe accuracy (%): Pass rate on 100 randomly sampled problems from 1,000-problem held-out pool

**NC_GUIWorld:**
- FVD (Frechet Video Distance): Distribution-level video quality
- SSIM: Per-frame structural similarity
- LPIPS: Learned Perceptual Image Patch Similarity
- Post-action variants (SSIM+15, LPIPS+15, FVD+15): Metrics averaged over the 15 frames following each click/scroll/key event
- Cursor accuracy (%): Fraction of frames with correctly placed cursor (Table 9)

### 1.5 Evaluation Protocol

- All evaluations open-loop: ground-truth conditioning sequences provided; generated frames not fed back
- CLIGen Clean: 5 uniformly sampled (ground-truth, generated) frame pairs per video for OCR evaluation
- Arithmetic probes: 100 random samples from 1,000-problem held-out pool; single draw per evaluation
- GUIWorld injection mode ablation: 64k training steps, 64 GPUs; all four modes share identical data and non-action components; each mode is a separate model
- GUIWorld data quality experiment: contextual conditioning fixed; only training data split varied
- Cursor supervision: SVG mask + reference frame vs. position-only vs. Fourier position encoding (Table 9)

## 2. Experiment-by-Experiment Assessment

### Experiment 1: NC Stays Readable at Practical Font Sizes

**Claim**: Wan2.1 VAE reconstructs terminal content well at 13px font but degrades at 6px.
**Table**: Table 2 (Average PSNR: 40.77, Average SSIM: 0.989 on CLIGen General frames).
**Variables isolated**: Font size.
**Assessment**: This is a prerequisite validation experiment, not an ablation — it tests whether the backbone is viable for terminal rendering before any NC-specific training. The result (PSNR governs by font size rather than content complexity) is the correct question to answer first.

**Weaknesses**:
- PSNR/SSIM averages can mask per-font-size variance. Table 2 reports a single aggregate average without breakdown by font size.
- No reconstruction comparison on GUI frames, so the VAE suitability check applies only to CLI content.
- The comparison to Rivard et al. (2025) claim (generic VAEs perform poorly on structured screenshots) is made qualitatively, not with reported numbers from that concurrent work.

### Experiment 2: Performance Plateaus Early and Can Degrade with Prolonged Training

**Claim**: PSNR/SSIM on CLIGen (Clean) plateau around 25k steps and extended training can slightly reduce them.
**Figure**: Figure 5(c) — PSNR/SSIM curves vs. training steps.
**Variables isolated**: Training step count.
**Assessment**: This is a learning curve analysis. The finding (early plateau) correctly motivates the 25k-step training recommendation for CLIGen (Clean). The paper's interpretation ("most learnable structured patterns are acquired early") is plausible.

**Weaknesses**:
- No reporting of absolute PSNR/SSIM values at each step in the text; Figure 5(c) is the sole evidence and its y-axis tick marks are small.
- No multiple-run variance is reported; it is unclear whether the plateau pattern is stable across different random seeds.
- The plateau behavior is only reported for CLIGen Clean; whether the same pattern holds for CLIGen General is not reported.
- Degradation at 460k steps is slight; the magnitude is not quantified in the text.

### Experiment 3: Literal Captions Drive Rendering Accuracy

**Claim**: Detailed captions improve PSNR from 21.90 dB (semantic) to 26.89 dB (detailed) — a ~5 dB gain.
**Table**: Table 3 (Caption styles vs. TI2V fidelity: semantic 21.90/0.813; regular 23.63/0.843; detailed 26.89/0.867).
**Variables isolated**: Caption specificity (semantic vs. regular vs. detailed).
**Assessment**: This is a well-designed ablation: the underlying terminal sequence is the same, only the caption tier changes. The monotonic progression from semantic to detailed provides convincing evidence that caption specificity is a meaningful conditioning lever.

**Weaknesses**:
- No error bars or confidence intervals are reported (single-point measurements per tier).
- The sample size for this comparison is not stated — how many clips/videos were used?
- The experiment is on CLIGen Clean only; whether the same gradient exists on CLIGen General is untested.
- The 5 dB PSNR difference between semantic and detailed captions is the paper's strongest single-variable result, making the absence of repeated-trial statistics a notable gap.

### Experiment 4: NCs Achieve Accurate Character-Level Text Generation

**Claim**: Character accuracy improves from 0.03 (at initialization) to 0.54 at 60k steps; exact-line accuracy reaches 0.31.
**Table**: Table 4 (OCR accuracy vs. training steps, with delta values from 0k baseline).
**Variables isolated**: Training step count.
**Assessment**: This is a training-progression tracking experiment. The delta notation (e.g., 0.54 ↑0.51) is a useful presentation convention showing absolute progress from initialization. The use of Tesseract OCR on paired frame pairs is a reasonable evaluation approach for terminal content.

**Weaknesses**:
- Tesseract is applied to generated frames without reporting its baseline accuracy on ground-truth frames. If Tesseract fails to read clean ground-truth terminal text in some cases, the 0.54 character accuracy understates the actual rendering fidelity.
- Only 5 uniformly sampled frame pairs per video are used for evaluation; this sampling introduces variance that is not quantified.
- No per-task-type accuracy breakdown: character accuracy is averaged over all CLIGen Clean content types, mixing regular traces (package installation, REPL) with Python math traces. These are qualitatively different and may have different fidelity profiles.
- Multiple-run statistics (standard deviation, confidence intervals) are absent.

### Experiment 5: Does This NC Instantiation Show Native CLI Reasoning?

**Claim**: Current video models including NC_CLIGen achieve low arithmetic accuracy (Wan2.1: 0%, NC_CLIGen: 4%, Veo3.1: 2%, Sora2: 71%).
**Table**: Table 5 (100 problems sampled from 1,000-problem held-out pool).
**Variables isolated**: Model identity.
**Assessment**: This is a comparative probe experiment. The held-out pool and random sampling approach are reasonable. The paper correctly interprets the low accuracy of all models except Sora2 as evidence of limited symbolic reliability in current video models.

**Weaknesses**:
- Sample size is small (100 problems from 1,000). No confidence interval is reported. At 4% accuracy (4/100 problems), a 95% Wilson CI spans roughly 1.1%–9.9%, making the distinction between NC_CLIGen (4%) and Wan2.1 (0%) statistically fragile.
- The generation configuration for Sora2 (temperature, number of sampling steps, system prompt, input format) is not reported. Sora2 at 71% is a substantial outlier. Three hypotheses are offered (Table 6) but none are tested experimentally.
- Veo3.1 achieves 2% while Wan2.1 achieves 0% — the paper does not discuss whether this difference is meaningful given the sample size.
- The held-out problem pool is described as "basic mathematical operations through terminal interactions" but the exact operations (addition, multiplication, multi-step?) are not specified, making difficulty calibration impossible.
- No attempt is made to test whether the 4% NC_CLIGen accuracy is above chance for a random-generation baseline.

### Experiment 6: Does This NC Instantiation Require RL for Symbolic Probes?

**Claim**: Reprompting (providing the answer in the caption) raises NC_CLIGen arithmetic accuracy from 4% to 83% (Figure 6).
**Variables isolated**: Conditioning signal (standard caption vs. answer-included reprompting).
**Assessment**: This experiment is correctly framed. The paper explicitly states: "we include the correct answer explicitly in roughly half of the training captions" and "we do not treat it as direct evidence of native arithmetic inside the NC backbone." This methodological distinction is appropriate and reflects intellectual honesty. The reprompting experiment tests steerability, not reasoning.

**Weaknesses**:
- The reprompting result (83%) is higher than Sora2 (71%) on the same probe set, but the paper does not directly compare them or note this. If reprompting provides a strong enough hint, this comparison loses discriminative value.
- The "roughly half" of training captions that include the answer introduces a training set distribution shift from the standard model. This means the reprompted model is a different trained model, not the same model with a different input at test time. This distinction is important and is not prominently flagged.
- No statistical significance testing for the 4% → 83% jump.

### Experiment 7: Data Quality Dominates Performance

**Claim**: CUA supervised data (110h) achieves FVD 14.72 vs. Random Fast (FVD 48.17, 400h) and Random Slow (FVD 20.37, 1,000h); untrained baseline FVD is 149.61.
**Table**: Table 8 (FVD, SSIM, LPIPS across data splits under contextual conditioning).
**Variables isolated**: Training data source (quality/goal-directedness vs. quantity).
**Assessment**: This is a well-structured dataset comparison. Holding the conditioning mode fixed (contextual throughout) and varying only the training data source isolates the data quality variable appropriately. The result (110h supervised >> 1,400h random combined) is a useful and practically actionable finding.

**Weaknesses**:
- Contextual conditioning is used throughout this experiment. It is not tested whether the data quality advantage holds under internal conditioning (the overall best-performing mode). If internal conditioning partially compensates for data quality, the magnitude of the quality effect may differ.
- CUA supervised data has significantly lower action density (0.10 actions/sec vs. 4.18 for Random Fast, Table 7). Action density is a confound: the supervised data may yield better FVD not only because it is goal-directed but also because sparser actions reduce motion artifacts. This confound is not controlled.
- Only global FVD/SSIM/LPIPS are reported; post-action metrics (which are more relevant to action-response quality) are not reported for this data comparison experiment.
- The app set and task distribution in CUA data may overlap more with the evaluation conditions than Random Fast/Slow. Selection effects are not discussed.

### Experiment 8: Precise Cursor Control Requires Explicit Visual Supervision

**Claim**: Position (x,y) only achieves 8.7% cursor accuracy; adding Fourier encoding raises it to 13.5%; SVG mask + reference frame achieves 98.7%.
**Table**: Table 9 (Loss variant vs. cursor accuracy).
**Variables isolated**: Cursor supervision modality (coordinate-only vs. Fourier position vs. visual mask+reference).
**Assessment**: This is the paper's most decisive single-variable ablation. The 8.7% → 98.7% gap between coordinate-only and visual supervision is large enough to be convincing without statistical testing. The three-way comparison (position, Fourier, visual) cleanly shows that the jump requires rich spatial reference, not just better position encoding.

**Weaknesses**:
- Cursor accuracy is a binary metric (correct/incorrect per frame) with no spatial precision breakdown. At 98.7% accuracy, it is unknown whether the 1.3% error cases are random or systematically distributed (e.g., during rapid cursor movements in Random Fast data).
- The evaluation is on the full GUIWorld dataset (all splits combined?), or only a specific split — not specified.
- No ablation testing whether the cursor accuracy advantage transfers to improvements in SSIM/FVD. The cursor accuracy is reported in isolation; its effect on the downstream task metrics (Table 8, Table 10) is not cross-referenced.

### Experiment 9: Action Injection Under Different Schemes

**Claim**: Internal conditioning achieves SSIM+15: 0.863 and FVD+15: 14.5, outperforming contextual (0.813/24.8), residual (0.857/18.8), and early-stopped external (0.746/33.4).
**Table**: Table 10 (Action-driven metrics across injection schemes, 15 frames after action).
**Variables isolated**: Injection mode (external, contextual, residual, internal).
**Assessment**: This is the central architectural ablation for NC_GUIWorld. The use of post-action metrics (SSIM+15, LPIPS+15, FVD+15) rather than overall metrics is appropriate: action-conditioning quality should be judged by how well the model responds to specific action events, not global video quality.

**Weaknesses**:
- External conditioning is early-stopped at ~50% of the planned training budget (acknowledged in footnote). This makes external vs. internal a confounded comparison: the performance difference could reflect training compute rather than architecture. The footnote correctly flags this, but it means the four-mode comparison has only three fully trained data points.
- The parameter count confound (HIGH-003 in CFR): internal mode adds CA_action sub-layers inside transformer blocks that require training, while contextual mode modifies only the attention mask. Observed SSIM differences (internal 0.863 vs. contextual 0.813) may partly reflect trainable parameter capacity.
- Residual mode achieves the best LPIPS (0.138), lower than internal (0.141), while internal achieves better SSIM and FVD. The paper recommends "internal" as the best overall mode, but the LPIPS difference indicates a trade-off. No criterion for combining SSIM, LPIPS, and FVD into an overall ranking is defined.
- No statistical significance testing. Given that SSIM differences (0.813 vs. 0.863) are small absolute numbers, their statistical significance is unknown.

### Experiment 10: Do Action Encodings Matter?

**Claim**: Meta-action (API-like) encoding achieves SSIM+15: 0.863, FVD: 14.5 vs. raw-action (event-stream): SSIM+15: 0.847, FVD: 16.6 — modest but consistent improvement.
**Table**: Table 11 (Raw-action vs. API-like encoding under internal conditioning).
**Variables isolated**: Action encoding representation (raw multi-hot vs. meta-action typed schema).
**Assessment**: This experiment correctly holds the injection mode fixed (internal) and varies only the encoding representation. The result — modest gains for meta-action encoding — is correctly interpreted: "encoding granularity is not the dominant factor governing GUI interaction fidelity."

**Weaknesses**:
- Only tested under internal conditioning; it is untested whether raw vs. meta-action differences would be larger or smaller under other injection modes (e.g., contextual, where actions are tokenized differently).
- The SSIM gain (0.847 → 0.863) and FVD gain (16.6 → 14.5) are modest. Statistical significance is not reported.
- Table 12 shows example encodings but does not quantify the token-length or embedding-dimensionality difference between the two encoders, making it impossible to assess whether meta-action's gains partially reflect a more compact representation rather than better semantics.

## 3. Dataset Appropriateness and Coverage

### Current State

**CLIGen datasets** are appropriate for the CLI-specific NC evaluation scope. CLIGen General covers diverse, real-world terminal use including package managers, text editors, REPL sessions, and system monitoring — sufficient for evaluating whether basic terminal "physics" are learnable. CLIGen Clean's scripted Docker traces provide the controlled, deterministic sequences needed for quantitative metric evaluation. The distinction between the two subsets is a methodological strength.

However, both datasets are drawn exclusively from Linux terminal environments (asciinema records primarily Linux/macOS developer workflows; the Docker containers are unspecified OS but implied Linux). Windows CMD/PowerShell and macOS-specific terminal behaviors are absent. The data is also heavily biased toward text-centric terminal use; interactive terminal UIs (e.g., ncurses, vim, tmux multi-pane) receive qualitative mention but no dedicated evaluation.

**GUIWorld dataset** is appropriate for the controlled evaluation of GUI interaction primitives. Ubuntu 22.04 / XFCE4 / 15 FPS at 1024x768 is a reasonable standardized environment. The fixed app set (Firefox ESR, GIMP, VLC, VS Code, Calculator, Terminal, File Manager, Mahjongg) covers a reasonable range of interaction types: web browsing, image editing, video playback, code editing, arithmetic, file management, games.

However, evaluation is restricted to a single OS/desktop-environment combination. No Windows, macOS, or mobile interface is included. The fixed app set excludes productivity applications (spreadsheets, word processors), communication tools (chat, email clients), and settings/system panels, which are common in real desktop usage. The controlled Ubuntu container environment also means visual appearance variability (themes, icon packs, font rendering) is minimal.

### Development Directions

- Add evaluation on at least one alternative OS/terminal environment (e.g., macOS zsh or Windows PowerShell for CLIGen; Windows 10 or macOS for GUIWorld) to test distributional generalization.
- Include at least one interactive full-screen terminal UI (vim, tmux) in the CLIGen Clean evaluation subset.
- For GUIWorld, add evaluation on a second desktop environment (e.g., KDE Plasma or macOS) to test whether results are specific to XFCE4 appearance conventions or general to desktop GUI interaction.
- Document the asciinema data distribution (top terminal commands, session length distribution, language composition) to allow readers to assess training data representativeness.

## 4. Baseline Selection Assessment

### Current State

**NC_CLIGen baselines** in the arithmetic probe experiment (Table 5) include Wan2.1 (the backbone), Veo3.1, and Sora2. These are reasonable comparators for the specific purpose of demonstrating that symbolic failure is widespread across current video models.

However, no text-to-video baseline specialized for screen content is included. NeuralOS (Rivard et al., 2025) is the closest architectural precedent for NC_CLIGen, yet it is not evaluated on the same probe set. The paper references NeuralOS only in passing in the Introduction, without a quantitative comparison. This is the most significant missing baseline for NC_CLIGen.

For perceptual quality evaluation (PSNR/SSIM in Tables 2–4), no baseline is compared: results are reported for NC_CLIGen only, with the untrained Wan2.1 serving as an implicit anchor only for Table 2 (VAE reconstruction). There is no comparison against any other fine-tuned model.

**NC_GUIWorld baselines** consist entirely of ablation conditions within the paper's own design space (untrained, external early-stopped, contextual, residual, internal). No external system — such as NeuralOS, Genie 3 (Bruce et al., 2024), or any prior GUI world model — is evaluated on the GUIWorld data. This means all GUIWorld results are self-referential: they demonstrate relative improvements within the paper's design choices, but do not establish NC_GUIWorld's performance level against the existing state of the art in interactive world models.

Genie 3 (Bruce et al., 2024) is cited in the Introduction as the closest related system ("action-controllable generative environments"), making it the natural baseline for NC_GUIWorld's post-action metrics. Its absence from the quantitative comparison is a notable gap.

### Development Directions

- Compare NC_CLIGen against NeuralOS (Rivard et al., 2025) on a shared evaluation set (either CLIGen Clean or a standardized terminal interaction benchmark) to quantify the NC_CLIGen contribution relative to the most similar prior system.
- Compare NC_GUIWorld against Genie 3 (Bruce et al., 2024) or Matrix-Game-2 (He et al., 2025) on post-action metrics (SSIM+15, FVD+15). If these systems cannot be directly run on GUIWorld data, at minimum report their performance on the authors' own evaluation datasets and discuss comparability.
- For arithmetic probes (Table 5), report a random-baseline accuracy estimate (expected accuracy if model generates plausible-looking but incorrect text) to contextualize the 4% NC_CLIGen result.
- If Sora2's 71% arithmetic accuracy is to be reported, include its generation configuration (temperature, prompt format, sampling steps) to allow meaningful comparison with NC_CLIGen's 4%.

## 5. Evaluation Metrics Assessment

### Current State

**NC_CLIGen metrics** span pixel-level quality (PSNR, SSIM), character-level fidelity (OCR character accuracy, exact-line accuracy), and functional capability (arithmetic probe accuracy). This multi-level hierarchy is appropriate: PSNR/SSIM capture rendering fidelity; OCR-based accuracy captures usability of generated text; arithmetic probes capture reasoning capability.

The OCR evaluation using Tesseract is reasonable for terminal content but carries compounding noise from two sources: (a) generation quality and (b) Tesseract reliability on generated frames. Since Tesseract's accuracy on clean ground-truth terminal frames is not reported, the absolute OCR numbers (0.54 character accuracy at 60k steps, Table 4) cannot be cleanly interpreted. The noise floor is unknown.

LPIPS is absent from the CLIGen evaluation. For perceptual quality of text rendering (where pixel alignment matters more than perceptual semantics), PSNR and SSIM are reasonable choices and LPIPS would add limited information.

**NC_GUIWorld metrics** use FVD, SSIM, and LPIPS globally plus post-action variants (SSIM+15, LPIPS+15, FVD+15). The post-action variant is a genuine contribution: evaluating the 15 frames immediately following each discrete action event focuses the metric on action-response quality rather than overall visual coherence. This is more relevant to the NC control primitive claim.

However, FVD measures distributional video quality, not frame-to-frame deterministic accuracy. In a setting where actions should produce specific, predictable interface transitions (e.g., clicking a button should produce a specific UI change), a distributional metric may not capture failure cases where the correct transition is generated with low probability. No deterministic evaluation (e.g., "did the button click produce the expected state in at least X% of samples?") is provided.

No task-completion or semantic correctness metric is reported for GUIWorld. SSIM and FVD measure perceptual quality, not whether the generated frames represent a correct response to the user's intended action.

### Development Directions

- Report Tesseract OCR accuracy on ground-truth terminal frames (CLIGen Clean evaluation set) to establish the measurement noise floor for character accuracy and exact-line accuracy in Table 4.
- Add a semantic correctness metric for GUIWorld: for a subset of CUA-supervised interactions (e.g., "open Firefox"), score whether the generated frames contain the target UI element (using template matching or VQA) in addition to SSIM/FVD.
- For arithmetic probes (Table 5), report multiple samples (e.g., 3-5 random draws of 100 from the 1,000-problem pool) to quantify sample-size variance and support confidence intervals.
- Consider adding LPIPS to the CLIGen rendering quality evaluation for comparison with GUIWorld and for completeness.

## 6. Experimental Rigor Assessment

### 6.1 Statistical Validity

**Current State**: The paper reports no confidence intervals, standard deviations, or significance tests for any of its 10 experiments. All results are single-point estimates. This is consistent with the systems-paper norm for computationally expensive experiments (22k–38k H100 GPU hours per system makes multiple full runs cost-prohibitive), but it affects the interpretability of small differences.

Specific cases where missing statistics are consequential:
- Table 3 (5 dB PSNR gap, 3-tier comparison): The magnitude is large enough to be robust, but the number of evaluation samples is not stated.
- Table 10 (SSIM internal 0.863 vs. contextual 0.813): A 0.050 SSIM difference may or may not be statistically significant; it cannot be assessed from the paper alone.
- Table 5 (arithmetic probe at 100 samples): NC_CLIGen 4% vs. Wan2.1 0% — the 95% Wilson CI for 4/100 overlaps with 0/100; the difference may not be statistically significant.
- Table 4 (OCR accuracy with delta notation): Multiple-run variance is absent; single-seed training curves are reported.

**Development Directions**: For the highest-value results (Table 3, Table 10, Table 5), report at least one of: (a) bootstrap confidence intervals from the evaluation set, (b) results from 2–3 training runs with different random seeds for ablations, or (c) explicit statement that results are single-run due to compute constraints. This last option at minimum manages reader expectations.

### 6.2 Reproducibility

**Current State**: The paper provides above-average reproducibility documentation for a systems paper. Training hyperparameters (AdamW lr=5e-5, weight decay 1e-2, bfloat16, gradient clipping 1.0, dropout 0.1) are stated. Hardware (64 H100 GPUs, ~15 days) and approximate compute (22k–38k GPU hours) are given. The appendix provides action schema, temporal alignment formulas, cursor rendering pipeline, and OCR evaluation protocol.

Gaps identified (carried forward from Phase 3-2, MEDIUM-001 and MEDIUM-002):
- Temporal window size w and lag l for contextual attention mask: not reported
- Contrastive loss temperature tau: not reported
- Layer indices for residual and internal injection: not specified
- CLIGen Clean filtering criterion: not described
- Action module hidden dimensions: not reported

**Development Directions**: Add a single configuration table in the appendix (or a supplementary config file) listing all unreported hyperparameters. This requires no new experiments and directly resolves the reproducibility gap.

### 6.3 Ablation Study Coverage

**Current State**: The paper includes five distinct ablations:
1. Caption specificity (Experiment 3): semantic vs. regular vs. detailed
2. Training step count (Experiments 2, 4): learning curves
3. Training data source (Experiment 7): random slow/fast vs. supervised
4. Cursor supervision modality (Experiment 8): position vs. Fourier vs. visual
5. Injection mode (Experiment 9): external vs. contextual vs. residual vs. internal
6. Action encoding representation (Experiment 10): raw vs. meta-action

This is a strong ablation portfolio for a systems paper. Each ablation cleanly isolates one variable at a time (when fully trained comparisons are available).

Gaps:
- No ablation over the number of action slots S in the meta-action encoder (S=2 is used without justification, per Phase 3-2 finding)
- No capacity-controlled injection mode comparison (HIGH-003 in CFR)
- No ablation over temporal window parameters (w, l) for contextual conditioning
- No ablation over training objective (diffusion loss only vs. diffusion + contrastive loss): the contribution of the temporal contrastive loss is not isolated

**Development Directions**:
- Add ablation over action slot count S (S=1, 2, 4) in meta-action encoder
- Add parameter-count-controlled injection mode comparison (match total trainable parameters across external/contextual/residual/internal)
- Report ablation removing the temporal contrastive loss to isolate its contribution to cursor accuracy and action-response quality

### 6.4 Hyperparameter Tuning Fairness

**Current State**: Section 3.2.3 states that all injection mode runs share "the backbone and all non-action components fixed" and "tune only the action encoder and learning-rate schedule." This implies separate learning rate searches per injection mode, which is appropriate. The training duration (64k steps) is uniform across modes, which is fair.

However, the external conditioning baseline is early-stopped at ~50% training compute (Table 10 footnote). This creates an asymmetric comparison: external vs. internal involves ~2x compute for internal, making the architecture comparison partially confounded with training budget. The paper correctly acknowledges this with the footnote, but does not report the external mode's extrapolated performance or an estimate of its fully-trained level.

**Development Directions**: Report a simple extrapolation of the external conditioning learning curve (using Figure 5 as a template) to estimate its likely fully-trained FVD/SSIM, and bound the extent to which early stopping accounts for the external-vs.-internal performance gap.

## 7. Experimental Comprehensiveness Assessment

### 7.1 Coverage of Primary Claims

The paper makes four explicit contributions:

**Claim 1**: "Define neural computers and build video-based prototypes for both CLI and GUI interfaces."
- Evidence: Two systems (NC_CLIGen, NC_GUIWorld) are built and described in detail. Experiments 1–10 provide quantitative characterization.
- Coverage: Adequate for a prototype definition paper.

**Claim 2**: "Provide a data engine and alignment recipe that synchronize text, actions, and frames for CLI and GUI."
- Evidence: Experiments 1 (VAE suitability), 2 (training curve), 7 (data quality), 8 (cursor supervision) provide direct evidence.
- Coverage: Solid. The data engine contribution is validated quantitatively.

**Claim 3**: "Identify practical design choices for NCs through extensive ablation studies."
- Evidence: Experiments 3, 9, 10 provide design-choice evidence. "Extensive" is slightly oversold for 10 experiments across two systems with some lacking full training runs (external baseline), but the ablations are appropriately structured.
- Coverage: Moderate. Ablations are clean but not exhaustive; several design choices lack ablations (S, w, l, contrastive loss weight).

**Claim 4**: "Outline an engineering roadmap toward completely neural computers (CNCs)."
- Evidence: Section 4 (position section), CNC definition, Table 14. This is a position/roadmap contribution, not validated by experiments.
- Coverage: The roadmap is well-articulated but is not experimentally grounded; it is a forward-looking position.

### 7.2 Missing Experiments

The following experiments would strengthen the paper's claims:

**Missing Experiment A: Closed-Loop Evaluation**
The paper acknowledges open-loop evaluation as a limitation (Section 3.2.4). Even a limited closed-loop test (e.g., 10 scripted CLIGen Clean tasks where generated frames are fed back as context for the next step) would provide the first evidence of whether short-horizon NC runtime behavior holds beyond single-clip open-loop rollout. This is the most important missing experiment given the paper's framing of NC as a "running computer."

**Missing Experiment B: Cross-Distribution Generalization for GUIWorld**
GUIWorld is evaluated on a fixed Ubuntu/XFCE4 environment. A simple test on a visual variant (e.g., different GTK theme or icon set) would show whether NC_GUIWorld has learned abstract GUI control primitives or is sensitive to specific visual appearances.

**Missing Experiment C: Temporal Contrastive Loss Ablation**
Training uses diffusion loss + temporal contrastive loss. The contribution of the contrastive loss is never isolated. An ablation (train without contrastive loss; report cursor accuracy, SSIM+15) would validate whether this training ingredient is responsible for the cursor accuracy improvement or whether explicit visual supervision alone suffices.

**Missing Experiment D: Multi-Step Action Accuracy for GUIWorld**
All GUIWorld evaluation uses single post-action windows (15 frames after one action). No multi-step evaluation is reported (e.g., accuracy over a 5-action sequence). This limits assessment of whether NC_GUIWorld maintains action-response accuracy over sequential interactions.

**Missing Experiment E: OCR Baseline on Ground-Truth Frames**
Report Tesseract character accuracy on ground-truth CLIGen Clean frames (before any NC generation). This calibrates the measurement noise floor and allows the observed 0.54 character accuracy (Table 4) to be interpreted as a fraction of the maximum achievable measurement.

## 8. Experimental Design Issues

### Issue 1: Open-Loop-Only Evaluation (CRITICAL for NC Claims)

**Description**: All 10 experiments use open-loop rollout — the model receives ground-truth conditioning sequences and generates video, but generated frames are never fed back as context. This is an inherent mismatch with the paper's central claim that NCs act as "running computers" with persistent latent runtime state h_t.

**Technical reasoning**: In a real NC deployment, the model would consume its own generated frames as x_t inputs to update h_t via F_theta, then generate the next x_t+1 via G_theta. The open-loop protocol provides ground-truth x_t at each step, bypassing any error accumulation in the latent state. The gap between open-loop metrics and closed-loop behavior is not just modest quantitative degradation — for a system claiming runtime state persistence, it represents an untested core property.

**Impact**: The paper's quantitative results (PSNR, SSIM, FVD, OCR accuracy, cursor accuracy) measure how well the model generates the next video frame given correct inputs, not how well it maintains coherent runtime state across self-generated frame sequences. The 10 experiments provide evidence of I/O alignment capability, not of runtime state persistence. The paper itself frames this correctly as a limitation, but no experiment provides even a single closed-loop data point.

**Status**: Newly identified as warranting HIGH severity in CFR (CFR-004).

### Issue 2: Early-Stopped External Baseline Conflates Architecture and Training Budget

**Description**: The external conditioning mode in Table 10 is trained to only ~50% of the planned budget. The footnote acknowledges it as "a rough reference," but the paper uses this comparison as part of its argument that deeper injection modes are architecturally superior.

**Impact**: If external conditioning's fully-trained performance is substantially better than its 50%-trained performance (consistent with Figure 5 showing continued improvement up to ~25k steps), the SSIM gap (external 0.746 vs. internal 0.863) could shrink significantly. The training compute asymmetry prevents a clean architectural conclusion from this comparison.

**Technical reasoning**: Figure 5 shows that CLIGen training curves still improve substantially between 10k and 25k steps. If an analogous pattern holds for GUIWorld injection mode training (which uses 64k steps as the full budget), an early stop at 32k could miss substantial improvement.

**Status**: Acknowledged in paper as caveat; no new CFR entry needed beyond existing HIGH-003.

### Issue 3: Arithmetic Probe Sample Size Too Small for Statistical Reliability

**Description**: Table 5 reports accuracy on 100 randomly sampled problems from a 1,000-problem pool. With 4/100 correct for NC_CLIGen, the 95% Wilson confidence interval is approximately [1.1%, 9.9%]. The difference from Wan2.1's 0/100 is statistically indistinguishable from zero at conventional significance thresholds.

**Impact**: The specific claim that NC_CLIGen (4%) demonstrates better symbolic capability than the base Wan2.1 model (0%) is not statistically supportable at n=100. The more robust claim — that all current video models except Sora2 fail on arithmetic probes — is well-supported regardless.

**Status**: Newly identified; severity MEDIUM (affects one claim within Experiment 5; does not invalidate the overall finding about video model limitations).

### Issue 4: Action Density Confound in Data Quality Experiment

**Description**: CUA supervised data has 0.10 actions/sec while Random Fast has 4.18 actions/sec (Table 7, a 42x difference). Experiment 7 concludes that goal-directedness (quality) drives the FVD advantage of CUA data, but action density is a major covariate.

**Technical reasoning**: High-density action streams (Random Fast) produce rapid, overlapping state changes that are intrinsically harder to model accurately. CUA data's lower action density means the model needs to predict accurate transitions for only one action per 10 seconds on average, compared to four actions per second for Random Fast. The FVD advantage of CUA (14.72 vs. 48.17) could partly reflect easier modeling of sparse action sequences rather than better action-response alignment.

**Status**: Newly identified; severity MEDIUM (affects interpretation of Experiment 7's conclusion about data quality; does not invalidate that CUA data is practically superior).

### Issue 5: No Cross-System Evaluation Protocol

**Description**: NC_CLIGen and NC_GUIWorld use entirely different evaluation metrics and there is no common evaluation framework. CLIGen uses PSNR, SSIM, OCR; GUIWorld uses FVD, SSIM, LPIPS + post-action variants. SSIM is the only shared metric, but it is computed on different data types under different conditions and the values are not compared.

**Impact**: The paper makes unified claims about "early NC primitives" for both systems, but the evidence is reported in separate, incommensurable metric spaces. It is impossible to assess whether NC_CLIGen's SSIM (Table 3, best 0.867 at detailed captions) and NC_GUIWorld's SSIM (Table 10, best 0.863) reflect similar or very different performance levels relative to their respective tasks.

**Development Direction**: Add at least one common metric computed under comparable conditions for both systems — for example, pixel-level SSIM on held-out non-action frames from both datasets, providing a shared reference point.

## 9. Validity Threats

### Internal Validity

**Threat 1: Evaluation set leakage for arithmetic probes**
The 50k Python math validation traces in CLIGen Clean overlap in domain with the arithmetic probes. If any training examples include problems from the same generation distribution as the held-out probe pool, the 4% NC_CLIGen accuracy may be inflated relative to a truly held-out symbolic reasoning test. The paper does not describe whether the arithmetic probe pool was constructed independently of the training data.

**Threat 2: Sora2 access and evaluation conditions**
Table 5 compares NC_CLIGen against Sora2 (71%), but Sora2's generation conditions (temperature, prompt format) are unreported. If Sora2 receives a more informative prompt or is sampled differently, its advantage is not cleanly attributable to model capability. The paper raises this concern in Table 6 (three hypotheses) but the comparison stands uncaveated in Table 5 itself.

**Threat 3: OCR evaluation noise as unknown floor**
The absolute OCR accuracy values in Table 4 (0.54 character accuracy) reflect compound noise from generation quality and Tesseract accuracy on generated frames. Without the Tesseract-on-ground-truth baseline, it is unknown what fraction of the apparent 46% error rate reflects generation failure vs. measurement noise.

### External Validity

**Threat 4: Single environment generalization**
Both NC_CLIGen and NC_GUIWorld are trained and evaluated on a single environment configuration (asciinema-sourced Linux terminal data; Ubuntu 22.04 XFCE4 at 1024x768 15 FPS). Generalization to other OS versions, desktop environments, themes, or screen resolutions is entirely untested.

**Threat 5: Fixed app set in GUIWorld**
The 7-app set (Firefox, GIMP, VLC, VS Code, Calculator, Terminal, File Manager, Mahjongg) does not cover the full breadth of desktop GUI interactions. Apps not in the training set are excluded from evaluation. The paper does not report any held-out-app evaluation, so whether NC_GUIWorld generalizes beyond its training app distribution is unknown.

## 10. Two-Layer Summary

### Current State

The experimental design is structured, clearly organized (10 numbered experiments, color-coded by system, each with a named takeaway), and appropriately partitioned between prerequisite validation (Experiments 1, 2), ablation studies (Experiments 3, 7–10), and capability probes (Experiments 4–6). The use of post-action metrics (SSIM+15, FVD+15) for GUIWorld is a genuine methodological contribution, correctly focusing measurement on action-response quality. The intellectual honesty of the Experiment 6 framing (reprompting as steerability, not native reasoning) and the early-stopped baseline footnote in Table 10 are marks of experimental self-awareness uncommon in systems papers.

Five categories of design weakness are identified:

1. **Absence of external baselines for the main systems**: NeuralOS and Genie 3 are the natural comparators for NC_CLIGen and NC_GUIWorld respectively, but neither is quantitatively compared. This leaves all results self-referential — demonstrating improvements within the paper's design space but not against the existing state of the art.

2. **No statistical significance reporting**: None of the 10 experiments reports error bars, confidence intervals, or significance tests. For small absolute differences (SSIM 0.813 vs. 0.863 in Table 10; 4/100 vs. 0/100 in Table 5), statistical significance cannot be assessed.

3. **Open-loop-only evaluation**: All 10 experiments use open-loop rollout. For a paper claiming "early NC primitives" of a running computer, the absence of any closed-loop data point means the key property (runtime state persistence under self-generated inputs) is entirely unmeasured.

4. **Action density confound in data quality experiment**: The CUA data advantage in Experiment 7 cannot be cleanly attributed to goal-directedness because CUA data is also 42x sparser in action density than Random Fast data.

5. **Early-stopped external baseline confounds Experiment 9**: The injection mode ablation's key finding (internal >> external) rests partly on an external baseline trained at half the compute budget.

### Development Directions

1. **External baselines**: Evaluate NC_CLIGen against NeuralOS on a shared terminal rendering task; evaluate NC_GUIWorld against Genie 3 or Matrix-Game-2 on post-action metrics. If full re-evaluation is infeasible, report human-preference comparisons on a shared subset of visualizations.

2. **Statistical rigor**: For high-value ablations (Tables 3, 5, 10), report bootstrap confidence intervals from the evaluation set. For the arithmetic probe (Table 5), draw 5 independent 100-sample batches and report mean ± std to characterize sampling variance.

3. **Minimal closed-loop evaluation**: Implement a single closed-loop evaluation on 50–100 scripted CLIGen Clean tasks (3–5 autoregressive steps with self-generated frames as context). Even a qualitative characterization of error accumulation would fill the most critical gap in the NC runtime primitive claim.

4. **Action density control**: Add a subsampled Random Fast condition (randomly drop actions to match CUA's 0.10 actions/sec density) to isolate whether the CUA advantage is primarily about goal-directedness or action sparsity.

5. **Tesseract noise floor**: Report Tesseract accuracy on ground-truth CLIGen Clean frames as a single calibration number, enabling interpretation of Table 4 absolute values.

6. **Full external conditioning training run**: Complete the external conditioning training run (at 64k steps) and report final Table 10 metrics, even if only in the appendix, to resolve the training budget confound in Experiment 9.
